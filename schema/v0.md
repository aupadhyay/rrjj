# rrjj event schema v0

This document is normative for schema version `0`. A stream is UTF-8 NDJSON:
each line is one complete JSON object. Producers may add fields without changing
the version. Consumers MUST ignore unknown fields.

## Envelope

Every event has:

- `v`: integer `0`.
- `seq`: unsigned integer, starting at `0`, strictly increasing by one within a
  session. An event is assigned a sequence only when accepted by the sink.
- `session_id`: non-empty opaque string.
- `ts`: RFC 3339 UTC timestamp with millisecond precision.
- `type`: one of the event names below.
- `data`: the event payload.

IDs are opaque strings prefixed by kind: `op:`, `c:` (commit), `t:` (tree), and
`b:` (blob). Paths are `/`-separated, UTF-8 paths relative to a recorded root.

## Events

### `session_start`

Emitted exactly once at sequence 0, after the baseline snapshot has been
persisted. Data: `roots: string[]`, `baseline_op`, `baseline_tree`, `ignore:
string[]`, `rrjj_version`, `jj_lib_version`, and `jj_store_version`.

### `snapshot`

Data: `op`, `parent_op`, `commit`, `tree`, `changes`, `stats`, and `truncated`.
Each change has `path`, `kind`, and optional `blob`, `prev_blob`, `from`, and
`bytes`. `kind` is `added`, `removed`, `modified`, `renamed`, or
`mode_changed`. `stats` contains counts for all five kinds.

When the full list exceeds the producer's configured limit, `truncated` is true
and `changes` contains at most that limit; `stats` still covers the complete
diff. The referenced trees remain authoritative. v0 does not infer renames;
producers emit a removal and addition unless they have sound copy metadata.

### `touched_paths`

Optional audit aggregation from raw filesystem notifications. Data:
`paths: object[]`, `raw_events: u64`, `window_started_at`, and
`window_ended_at`. Each path object has a relative `path` and a sorted,
deduplicated `operations` array containing `create`, `modify`, `remove`,
`rename`, or `other`. Paths are sorted and deduplicated. Operations describe
notifications observed during the window; repeated writes are not byte-version
snapshots. This event is only audit context: snapshots, not notifications,
determine recorded state.

### `mark`

Data: `label`, `meta` (an arbitrary JSON object), and `ref_op`. A requested mark
MUST be emitted after any snapshot already pending in the coordinator.

### `session_end`

Best-effort lifecycle termination event. Data: `final_op`, `reason`,
`snapshots`, and `events`. `events` is the number of events preceding
`session_end`. A graceful shutdown emits one final `flush` immediately after
`session_end` so the termination event itself becomes reader-visible and
durable.

### `flush`

Data: `op` and `through_seq`. A durable sink publishes its reader-visible
manifest only after the event stream and every repository object needed to read
`op` have been stored. The manifest's `durable_seq` and `durable_op` are the
authoritative reader watermark.

### `error`

Data: `scope`, `message`, `fatal`, and `retrying`.

### `overflow`

Records loss of filesystem notifications. Data: `source`, `raw_events`, and
`recovery`. `recovery` is `full_scan_snapshot`. An overflow makes the workspace
dirty and MUST trigger a full-scan snapshot before a later mark is emitted.
Notify rescan requests and backend errors that may imply dropped notifications
are overflows.

## Session manifest

Durable local session stores publish a `manifest.json` with the session and
format metadata, observed `last_seq`/`last_op`, and optional
`durable_seq`/`durable_op` watermarks. `events_object`, when present, is the
relative path of the immutable NDJSON object containing events through the
durable watermark.

`storage` is optional so existing v0 manifests remain valid. When present, it
contains URI pointers for control-plane indexes. Remote Git + HTTP deployments
typically leave local `storage` unset; hosts persist opaque event JSON and the
`sequence → commit OID` association themselves.

Readers MUST treat the local manifest and its durable watermarks as
authoritative for directory sessions. Cold-storage pointers are locators, not
evidence that an upload completed.

### Remote HTTP event batches

Hosts that receive timeline events over HTTP should accept contiguous prefixes
keyed by `(session_id, seq)` and acknowledge:

- `accepted_through_seq`: highest contiguous accepted sequence in the request
- `durable_through_seq`: present on final durable batches; the watermark the
  producer may advance locally

Snapshot `data.commit` values use the `c:` prefix; the accompanying durable
`checkpoint` field is the canonical Git OID hex published before the batch.

### Ordering for coordinated sinks

For a flush that includes a snapshot:

1. Accept events into the synced local spool.
2. Publish checkpoint Git objects and advance the recorder-owned ref.
3. Publish snapshot and subsequent events through the target sequence.
4. Advance the durable watermark only after acknowledgement.

A Git ref may become visible before event acknowledgement; that orphaned
checkpoint is unindexed until event publication succeeds. The opposite order is
not allowed.

## Ordering and recovery

One coordinator owns snapshotting, event projection, and sequence assignment.
This is the schema's single-writer requirement. Missing sequence numbers imply
loss; repeated or decreasing numbers imply duplication or reorder. Absence of
`session_end` is allowed after an ungraceful termination.
